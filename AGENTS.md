# Архитектура проекта: Polzuchie-tvari.io (Snake AI)

Проект представляет собой мультиплеерную игру "Змейка" в реальном времени с 3D-графикой на клиенте и высокооптимизированной симуляцией на сервере.

## Структура проекта

Проект разделен на две основные части: `backend` и `frontend`.

### Backend (Серверная часть)
Расположение: `/backend`
Стек: Python, FastAPI, Uvicorn, WebSockets, MessagePack, zlib.

- **`backend/server.py`**: Точка входа в приложение. Инициализирует FastAPI, настраивает CORS, управляет жизненным циклом сервера (`lifespan`), асинхронным игровым циклом (`game_loop`) с заданной частотой тиков и маршрутизацией запросов.
- **`backend/game_config.py`**: Управление конфигурацией игры на основе датаклассов (`WorldConfig`, `SimulationConfig`, `SnakeConfig`, `BoostConfig`, `FoodConfig`, `VisualConfig`).
  - Поддерживает строгую валидацию параметров при патчах и безопасную проверку динамических формул стоимости роста (`growth_score_per_segment`) через AST.
  - Содержит механизм обратной совместимости при обновлении схемы: неизвестные или устаревшие ключи настроек (например, старый блок `network`) пропускаются с предупреждением, а не вызывают критический сбой.
- **`backend/app/engine/state.py`**: Основной движок игры, реализующий класс `GameState`.
  - Управляет размерами поля, спавном еды, перемещением и физикой змеек.
  - Использует **Пространственное разделение (Spatial Partitioning)** для оптимизации коллизий и выборки видимых объектов с сеткой 10.0.
  - Рассчитывает видимость по радиусу **Area of Interest (AoI)**, который вычисляется динамически для каждого игрока на основе текущего радиуса видимого тумана (fog radius) с 3% буфером безопасности.
- **`backend/app/engine/entities.py`**: Описание сущностей игрового мира (`Snake`, `Food`, `BlackHole`, `Portal`).
- **`backend/app/engine/physics.py`**: Расчет физики движения, скоростей и векторов направления.
- **`backend/app/engine/collision.py`**: Логика коллизий. Внедрены физически точные столкновения (дистанционный коэффициент 0.95), исключающие просачивание змеек друг в друга.
- **`backend/app/api/websocket.py`**: Обработка WebSocket-подключений, поддержание пула активных соединений, отправка сжатых стейтов из очереди сообщений.
- **`backend/app/api/admin.py`**: API-эндпоинты панели администратора (получение, применение конфигурации, статус сервера и перезапуск).
- **`backend/requirements.txt`**: Зависимости (`fastapi`, `uvicorn[standard]`, `msgpack`, `websockets`).

### Frontend (Клиентская часть)
Расположение: `/frontend`
Стек: Next.js (App Router), React 19, Three.js / React Three Fiber, HTML5 Canvas.

- **`frontend/hooks/game.worker.ts` (Фоновый Web Worker)**:
  - Выполняется в отдельном потоке процессора. Управляет WebSocket-подключением, реконнектами и PING/PONG-запросами.
  - Декомпрессирует входящие бинарные MessagePack-пакеты, сжатые сервером методом `zlib`, используя встроенный браузерный API `DecompressionStream("deflate")`.
  - **Вычисления сплайнов**: Полностью берет на себя тяжелые расчеты 3D-сплайнов (координаты, нормали, сглаживание, сегментация на границах wrap-зон, анимация глаз/зрачков, частицы свечения хвоста).
  - Передает готовые расчеты на главный поток в виде **Transferable Objects** (`Float32Array` и `Uint32Array`), исключая накладные расходы на копирование памяти.
  - Синхронизируется с рендерером с помощью механизма подтверждения кадров (`isWaitingForFrameRef`).
- **`frontend/hooks/useGameSocket.ts`**:
  - Инициализирует веб-воркер, транслирует события ввода пользователя в сокет, парсит плоские массивы координат с проверкой на четность длины и NaN-безопасность (`parsePoints`), предохраняя WebGL от сбоев.
- **`frontend/components/GameRenderer.tsx`**:
  - **3D-Рендеринг**: Отображает мир через React Three Fiber.
  - Применяет преаллоцированные буферы и обновление атрибутов геометрии на месте (`updateDynamicAttribute`) без пересоздания объектов, сводя к нулю нагрузку на Garbage Collector и убирая микрофризы.
  - Реализует динамическую плотность сплайна в зависимости от длины змейки (score > 400: плотность 1 шаг/сегмент; иначе: 2 шага/сегмент), уменьшая количество полигонов на 75% для больших змей.
  - Выполняет Viewport Culling для отсечения невидимой еды и змеек вне зоны тумана игрока.
  - Отрисовывает 2D-радар Canvas с частотой 15 FPS для снижения нагрузки на мобильные GPU.
- **`frontend/components/game/`**: вспомогательные компоненты сцены.
  - **`materials.ts`**: Описание шейдеров и материалов. Все шейдеры материалов змеек и еды снабжены проверками `#ifdef USE_INSTANCING_COLOR` для предотвращения ошибок компиляции до инициализации цвета инстансов. Реализован GPU-клиппинг (`discard`) сегментов змеек, выходящих за границы карты (с сохранением плавного перехода головы на границе).
  - **`LensingEffect.tsx`**: Пост-процессинг проход, симулирующий гравитационное линзирование (эффект кольца Эйнштейна) вокруг черных дыр.
- **`frontend/app/page.tsx`**: Игровой HUD и сборщик пользовательского ввода (клавиатура, мышь с пропорциональной чувствительностью руления, гироскоп).
  - HUD обновляется напрямую в DOM в обход циклов ре-рендера React для минимизации задержек.
  - Компоновка интерфейса:
    - Симметричные отступы сверху для боковых панелей (`top: 20px`).
    - Слева внизу: Складывающаяся вверх панель управления (`Help Panel`) с динамически привязанным к ней списком набранных очков (`scoreFeed`).
    - Справа внизу: Текстовый индикатор пинга с высококонтрастной обводкой, расположенный над миникартой.
    - Справа вверху: Виджет "Топ 10" лидеров с ограничением по высоте и кастомным стеклянным скроллбаром.
    - Снизу по центру: Стеклянная шкала поворота мыши с прямым DOM-обновлением цвета и свечения указателя в зависимости от угла поворота.
- **`frontend/app/admin/`**: Панель администратора.
  - Переведена на английский язык.
  - Содержит интерактивный Canvas-симулятор распределения еды, визуализирующий настройки кластеризации (Box-Muller нормальное распределение) в реальном времени с поддержкой псевдослучайной генерации на базе сида (LCG) и драфтов изменений.

## Ключевые механизмы оптимизации и связи
1. **Формат данных и сжатие сети**: WebSocket-сообщения кодируются в MessagePack, координаты змеек упакованы в плоские массивы чисел `[x0, y0, x1, y1, ...]` (экономия трафика 35%), все вещественные числа округляются до 2 знаков после запятой (еще 25%). Пакеты сжимаются `zlib` на сервере и распаковываются через `DecompressionStream` в фоновом воркере на клиенте.
2. **Параллакс и позиционирование по высоте**:
  - Еда левитирует на высоте Z = 1.5, тело змейки — на Z = 2.0 (глаза Z = 2.1, зрачки Z = 2.11).
  - За счет угла наклона камеры 55 градусов создается естественный 3D-параллакс без искусственных сдвигов.
  - Еда проецирует мягкую размытую экспоненциальную тень (Gaussian shadow shader) на плоскость Z = 0.2 без эффекта обрезки краев.
  - Настройки `renderOrder` гарантируют корректное наложение слоев при переползании еды змейкой.
  - Радиус тени змейки имеет постоянный оффсет (+3.2 единицы) для предотвращения раздувания теней гигантских змей.
3. **Кэширование на сервере (MsgPack Caching)**: Сериализованные бинарные данные игроков кэшируются за тик и встраиваются в пакеты AoI через замену плейсхолдера, предотвращая лишние циклы сериализации на CPU.
4. **Ускорение (Буст)**: Удвоение скорости движения с потерей массы позади хвоста раз в секунду.
5. **Горячая перезагрузка сервера**: В режиме разработки watch-сервер Uvicorn настроен на полное игнорирование папок виртуального окружения `.venv` и кэша `__pycache__` для исключения зависаний дискового планировщика в ОС Windows.
