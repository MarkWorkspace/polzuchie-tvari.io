# Polzuchie-tvari.io — AGENTS

> Структура проекта — `PROJECT_MAP.md`. Архитектура и потоки данных — `ARCHITECTURE.md`. Статус фич — `GAME_STATE.md`.

---

## ⚠️ ПРАВИЛА

### 1. Лимиты размера
**Файл: максимум 300 строк. Функция: максимум 30 строк.**

### 2. Один файл — одна ответственность
Каждый файл делает ровно одну вещь. Название файла полностью описывает содержимое. God files запрещены.

### 3. Стек — без отклонений
**Frontend:** Vite + TypeScript + Three.js. Стили — только `styles/*.css`. Inline-стили объектами в JS/TS запрещены — допустимо только `el.style.opacity = value` для анимации.

**Backend:** Python + FastAPI (допускаются сторонние библиотеки). Новая механика — новый файл в `engine/systems/`.

### 4. Нет дублирования кода
Эти пары трогать только синхронно — изменение одного без другого ломает игру:

| Логика | Frontend | Backend |
|---|---|---|
| Тороидальная математика | `worker/shared/MathUtils.ts` | `engine/systems/math_utils.py` |
| Парсер формул роста | `worker/shared/FormulaParser.ts` | `game_config.py` |
| Сетевой формат кадра | `worker/DeltaDecoder.ts` | `engine/systems/serialization.py` |

**Примечание:** `game/Config.ts` содержит только клиентские константы рендеринга (gridSize, wrapOffsets, SKINS, WORLD_WIDTH/HEIGHT). Физические константы там более нет — клиент получает их от сервера через `server_simulation` в каждом кадре. Не добавлять физические константы в Config.ts.

Общий код для воркера — в `worker/shared/`.

### 5. Перед изменением
1. Прочитай `ARCHITECTURE.md` — найди затрагиваемый поток данных.
2. Прочитай `GAME_STATE.md` — проверь статус связанных фич и риски.
3. Если затрагивается 3+ файла — выведи список и изменения, жди подтверждения.

### 6. Backend — оркестратор без логики
Игровую логику в `game.py` не добавлять. Порядок выполнения систем определяется свойством `order` каждой системы — при добавлении новой системы в `engine/systems/` задай ей корректный `order` относительно существующих. Не меняй `order` существующих систем без явного указания.

### 7. Карта проекта — всегда актуальна
После создания, удаления, переименования или перемещения файла:
1. Убедиться, что первая строка файла содержит `ROLE:`-комментарий.
2. Обновить `PROJECT_MAP.md`.

Формат:
```typescript
// ROLE: Что делает. Что не делает (если граница неочевидна).
```

### 8. После задачи — обнови GAME_STATE.md
Отметить что исправлено, что добавлено, какие новые риски появились.

### 9. Тесты обязательны
Новые механики и исправления багов принимаются только если покрыты тестами. Тесты бэкенда — в `backend/tests/`. Формат: pytest. При исправлении бага сначала написать тест, воспроизводящий его, затем исправлять.
