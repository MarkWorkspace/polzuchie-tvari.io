# Архитектура проекта: Polzuchie-tvari.io (Snake AI)

Проект представляет собой мультиплеерную игру "Змейка" в реальном времени с 3D-графикой на клиенте и высокооптимизированной симуляцией на сервере.

## Структура проекта

Проект разделен на две основные части: `backend` и `frontend`.

### Backend (Серверная часть)
Расположение: `/backend`
Стек: Python, FastAPI, Uvicorn, WebSockets, MessagePack.

- **`backend/server.py`**: Точка входа и основная логика сервера. 
  - Реализует класс `GameState`, который управляет игровым полем (динамический размер в координатах сетки, базовый 100x100), спавном еды, физикой движения и столкновениями, используя оптимизацию **Пространственного разделения (Spatial Partitioning)** с размером ячейки сетки 10.0.
  - Игровой цикл (`game_loop`) работает асинхронно с частотой **20-30 тиков в секунду (tick rate)**. 
  - Для оптимизации сети внедрены **Area of Interest (AoI)** (клиент получает только то, что рядом) и **Настоящая Дельта (True Delta)** (вместо всего хвоста сервер шлет только координаты новых добавленных голов, упакованные в плоские массивы чисел `[x0, y0, x1, y1, ...]`).
  - Управление событиями клиентов (подключение, отключение, смена направления) происходит в эндпоинте `/ws/`.
  - Реализует REST эндпоинты `/admin/config` (GET и PATCH) для динамического изменения настроек игры администраторами.
  - Горячая перезагрузка (Hot-reload) сервера управляется переменной окружения `ENVIRONMENT` (отключается в продакшене). В режиме разработки Uvicorn и скрипт `start_dev.bat` настроены на исключение папок `.venv` и `__pycache__` из сканирования вотчера (`reload_excludes`), предотвращая фризы планировщика на Windows.
- **`backend/game_config.py`**: Система конфигурации игры на основе датаклассов (`WorldConfig`, `SimulationConfig`, `SnakeConfig`, `BoostConfig`, `FoodConfig`, `VisualConfig`).
  - Обеспечивает строгую валидацию диапазонов значений всех параметров при применении патчей.
  - Поддерживает динамические формулы стоимости роста сегмента (`growth_score_per_segment`), проверяемые через абстрактное синтаксическое дерево (AST) для предотвращения выполнения небезопасного кода.
  - Синхронизируется с файлом `config.json` на диске для персистентного хранения настроек.
- **`backend/requirements.txt`**: Зависимости (`fastapi`, `uvicorn[standard]`, `msgpack`, `websockets`).

### Frontend (Клиентская часть)
Расположение: `/frontend`
Стек: Next.js (App Router), React 19, Three.js / React Three Fiber (WebGL-рендерер), HTML5 Canvas (для радара и админки).

- **`frontend/hooks/game.worker.ts` (Фоновый Web Worker)**:
  - Выполняется на отдельном логическом ядре процессора, полностью разгружая главный поток.
  - Управляет жизненным циклом WebSocket-соединения, автоматическими экспоненциальными переподключениями и расчетом пинга в цикле PING/PONG.
  - Принимает бинарные MessagePack-пакеты от сервера, декодирует их и осуществляет пошаговую сборку игрового состояния (накапливает DELTA-патчи игроков и еды в FULL-стейт, обрабатывает интерполяцию магнитной еды), передавая на главный поток только готовые отфильтрованные данные.
- **`frontend/hooks/useGameSocket.ts`**:
  - Инициализирует Web Worker и принимает от него готовые обновления `GameState`.
  - Предоставляет mock-обертку сокета (`socketRef.current`), имитирующую методы нативного WebSocket, обеспечивая совместимость с системами ввода в `page.tsx` и `GameRenderer.tsx`.
- **`frontend/app/page.tsx`**: Основной интерфейс игры и HUD.
  - Обрабатывает ввод пользователя: клавиатурное управление (A/D/Стрелочки), мышь (аналоговое отклонение) и гироскоп (наклон телефона в мобильной версии) через фоновые window-слушатели.
  - **Реактивно-независимый HUD (React-less updates)**: Во время активного геймплея исключены любые ре-рендеры React страницы `page.tsx`. Очки накапливаются и вставляются в DOM точечно через Vanilla JS, пинг обновляет текстовый узел по ID, а таблица лидеров вынесена в самообновляющийся по кастомным событиям (`game-leaderboard-update`) компонент `Leaderboard.tsx`.
- **`frontend/components/GameRenderer.tsx`**:
  - **Рендеринг (WebGL/Three.js)**: Отрисовывает 3D-сцену через React Three Fiber. Для эффекта 3D применяется наклон камеры на 55 градусов, а для высокой производительности все змеи рендерятся в едином GPU-буфере динамического меша (BufferGeometry) без переаллокаций памяти (BufferAttribute-обновления «на месте»), что минимизирует сборку мусора (GC).
  - **Миникарта/Радар**: Радар рисуется на классическом 2D Canvas с принудительным ограничением фреймрейта до 15 FPS для снижения нагрузки на GPU мобильных устройств.
- **`frontend/app/admin/`**: Панель администратора.
  - Предоставляет визуальный веб-интерфейс для мониторинга активных игроков, спавна еды и интерактивного редактирования параметров конфигурации сервера в реальном времени с авторизацией по паролю.
- **`frontend/package.json`**: Зависимости фронтенда (React 19, Next.js 16, Three.js 0.184, `@msgpack/msgpack`).

### Инфраструктура и Деплой (Docker & Nginx)
- **`docker-compose.yml` & `.env`**: Оркестрация контейнеров `backend` и `frontend`. Внешние порты привязываются строго к `127.0.0.1` (доступ извне идет только через Nginx).
- **Dockerfiles**: Сборка изолированных образов. Фронтенд собирается в максимально легковесном режиме `standalone` (указано в `next.config.mjs`).
- **Nginx (Reverse Proxy)**: В продакшене используется для SSL (HTTPS) и маршрутизации. Запросы на `/` идут на фронт (порт 3000), а пути `/ws/` проксируются на бэкенд (порт 8000).

## Ключевые механизмы и коммуникация
1. **Сетевой протокол:** WebSockets.
2. **Формат данных:** Используется бинарный формат **MessagePack**. Координаты змей пакуются на сервере в виде плоских списков float с точностью до 2 знаков после запятой (`[x0, y0, x1, y1, ...]`), снижая сетевой трафик на 35%.
3. **Кэширование на сервере (MsgPack Caching):** Сериализованные байты координат каждого игрока кэшируются после их генерации в начале тика, что снижает CPU-нагрузку при рассылке индивидуальных пакетов AoI десяткам подключенных клиентов.
4. **Механика коллизий:** Физически точные столкновения (дистанционный коэффициент 0.95), исключающие просачивание голов сквозь чужие тела. При гибели змейка оставляет 50% от своих очков в виде яблок (цвета яблок наследуют цвета скина погибшего), после чего моментально возрождается.
5. **Магнитное притяжение еды:** Еда в радиусе притяжения плавно стягивается к голове змейки, её перемещения передаются клиентам через массив `moved_foods` для плавной интерполяции.
6. **Ускорение (Буст):** При зажатом пробеле (или специальной кнопке буста на мобильном) змейка ускоряется в 2 раза, сбрасывая массу позади хвоста раз в секунду.
